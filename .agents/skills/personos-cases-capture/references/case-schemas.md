# PersonOS Case Schemas

## Common Rules

All cases require:

- `编号`: generated by `append_case.py`
- `状态`: `真实/已验证` or `真实/待验证`
- `发生日期`: `YYYY-MM-DD` when known; otherwise `待确认`
- `项目`: concrete project name; use `跨项目` only when appropriate
- `证据来源`: array of repository-relative paths or short artifact descriptions
- `待验证项`: array; use `[]` when nothing remains unresolved
- `标签`: concise array without `示例`
- `关联案例`: optional array of exact case IDs; use when a later event deserves a new case
  but materially follows from an existing case

Record observed facts separately from interpretation. Evidence proves that an artifact or
action exists; it does not automatically prove why the user chose it or whether it succeeded.

## Type Routing

| Type | Destination | Choose when |
| --- | --- | --- |
| `decision` | `04_cases/decision_cases.jsonl` | The event centers on choosing among alternatives |
| `failure` | `04_cases/failure_cases.jsonl` | A harmful result and learnable cause are central |
| `success` | `04_cases/success_cases.jsonl` | A beneficial result and reusable action are central |
| `project` | `04_cases/project_cases.jsonl` | It is a meaningful project episode without a stronger type |

Use one primary type only. Prefer decision over project when alternatives and rationale are
known. Prefer failure or success over decision when the result and lesson are the main value.

## Update Versus Related New Case

- Update the existing case when the new information resolves its `待验证项`, corrects its
  facts, or supplies the outcome of the same event.
- Create a related new case when the follow-up is a distinct later event with its own lesson.
- Do not change a case's primary type during update. Preserve its `编号`.
- An update input must be a complete replacement object, not a partial patch.
- When an update resolves all uncertainty, change `状态` to `真实/已验证` and set
  `待验证项` to `[]`.

## Required Fields By Type

### Decision

```json
{
  "状态": "真实/待验证",
  "发生日期": "2026-06-14",
  "项目": "PersonOS",
  "决策背景": "...",
  "候选方案": ["..."],
  "最终选择": "...",
  "选择理由": ["..."],
  "结果": "...",
  "复盘": "...",
  "证据来源": ["README.md"],
  "待验证项": ["..."],
  "标签": ["产品决策"]
}
```

决策记录另有一条与 `状态`（可信度）正交的生命周期轴，由 `append_case.py` 自动补齐，
无需在草稿里手填默认值：

- `决策状态`：`现行`（默认）/ `已取代` / `已废弃` / `提议中`。
- `取代`：本条取代了哪条旧决策的 `编号`（写在**新**决策上）；无则空串。
- `被取代`：本条被哪条新决策取代的 `编号`（写在**旧**决策上，由脚本自动写入）；无则空串。

**取代配对（关键）**：当一个新决策否定/替换了一条既有决策时，在草稿输入对象里给
`取代` 填上旧决策编号。写入时脚本会校验旧编号存在，追加新决策（`决策状态:现行`），
并把旧决策自动改成 `决策状态:已取代` + `被取代:<新编号>`。旧决策不删除，保留原始背景与理由。
仅当确实是替换关系时才填 `取代`；只是相关、延续或叠加时用 `关联案例`，不要填 `取代`。

### Failure

Required type-specific fields: `背景`, `失败表现`, `根因`, `预警信号`, `修正措施`,
`预防方式`.

### Success

Required type-specific fields: `背景`, `成功表现`, `关键动作`, `成功因素`,
`可复用经验`.

### Project

Required type-specific fields: `背景`, `目标`, `行动`, `结果`, `经验`.

## Status Policy

- Use `真实/已验证` only when the event, important rationale, and result are confirmed.
- Use `真实/待验证` if the result has not happened, the date is uncertain, rationale is
  inferred, or an important claim lacks evidence.
- Name every unresolved detail in `待验证项`; do not silently fill it.
