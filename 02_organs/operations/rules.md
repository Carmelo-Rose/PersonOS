### 规则名称：生产项目上线前必须先统一源码、数据、平台和备份边界

- 规则编号：待生成或待填写
- 状态：待验证
- 适用场景：新项目或既有项目准备长期运行在服务机、需要多人/多项目运维时
- 正向要求：在开放长期运行前，先固定源码根目录、运行数据根目录、结构化数据库、对象存储、凭据、日志、备份和启停入口；完成可回滚迁移和端到端验收后再清理旧路径。
- 应避免：不要让项目把 SQLite、上传文件、日志、模型缓存和密钥随意散落在源码目录，也不要在未确认回滚与健康检查前直接删除旧数据或旧路径。
- 判断依据：本次用户明确要求统一规划、要求以后可定位可运维，并批准在未到真正生产阶段清理旧数据；迁移后五个业务库、MinIO、业务服务和备份均通过验收。
- 示例：
- 本次将 project/workspace 合并到 apps，并把 PostgreSQL、MinIO、prod-data、ops、secrets 分层固定。
- 迁移前归档 SQLite，迁移后验证 PostgreSQL 表、MinIO 应用级上传读取、服务健康和备份，再删除空旧目录。
- 允许例外：浏览器 profile、GPU 模型权重、ComfyUI 等外部安装软件可保持独立目录，但必须在布局文档中登记边界；不可丢失数据还需外部/异地备份。
- 置信度：低
- 用户动作/判断来源：用户在本次服务机生产化整理中要求先解决项目目录分散和数据落点不清，再统一启动 PostgreSQL、MinIO 和备份体系, 本次迁移中先归档 SQLite、再切换数据库/对象存储、最后清理空旧目录的实际结果
- 标签：生产化, 部署, 数据迁移, 运维
- 待确认或待验证：该规则目前只有本次服务机迁移一组直接证据，需在其他服务机项目中重复验证后再提高置信度。

### 规则名称：内网专用分支的基础设施迁移要按实际使用收窄范围

- 规则编号：待生成或待填写
- 状态：待验证
- 适用场景：把某个内网/公司自用分支或部署的基础设施迁移到新环境时，探查过程中发现代码里还有更复杂、面向未来商业化/对外场景的功能（计费、配额、第三方解析等）
- 正向要求：只迁移当前内网分支实际会调用、实际在用的那部分功能；复杂的商业化功能集继续留在原有基础设施上不动，等真正需要（比如产品上市）时再单独评估迁移
- 应避免：不要因为顺便发现了更完整/更复杂的功能集，就扩大这次迁移的范围一起搬过去——这会显著放大工作量和风险，而且很可能不是当前这次任务真正需要的
- 判断依据：内网自用场景的真实需求和面向市场的商业化需求往往规模差一个量级；把范围收窄到实际需要，能用更小的改动、更低的风险达到用户当下的目的
- 示例：
- Mono 插件真实后端有积分/套餐/退款/抖音解析/服务端视频分析等一整套商业化逻辑，但用户只要求迁移 /activate /verify 这两个内网真正用到的接口，其余保留在 veFaaS 未动
- 允许例外：如果用户明确说要为将来商业化预先做好准备，或者明确要求把完整功能集一起迁移，则不适用本收窄原则
- 置信度：中
- 用户动作/判断来源：2026-07-22 Mono/Baoluo 激活码迁移：用户明确说 company 分支不考虑积分/套餐/退款，只在意激活码，那些是为将来上市准备的
- 标签：基础设施迁移, 范围控制, 内网/商业化分离
- 待确认或待验证：无

### 规则名称：编辑"生产源码"前先验证它就是当前实际活跃的版本

- 规则编号：待生成或待填写
- 状态：待验证
- 适用场景：一个项目在文件系统里存在多个同名/相似目录（旧版本、参考副本、构建产物、不同分支的导出），需要修改"生产在用"的那份代码
- 正向要求：动手改之前，先用可验证信号确认目标目录确实是当前活跃版本：查 git branch/status 是否是预期分支、内容里是否有 mock/占位标记而非真实值、文件是否是压缩成一整行的混淆产物（build.js 之类工具的输出特征）
- 应避免：不要仅凭目录名字、旧的 memory 记录或者"看起来像是这个项目"就直接假设它是当前活跃源码并开始编辑
- 判断依据：多份代码库并存时，目录命名和历史记忆很容易过时或误导；git 分支状态和代码里的具体特征（provider 是否 mock、文件是否被混淆）是更可靠的活跃性信号
- 示例：
- D:\谷歌插件版本\imageVedioToPrompt - claude 的 background.js 里 IMAGE2_PROVIDER="mock"，说明这是未接真实功能的参考副本，不是生产源码
- D:\谷歌插件版本\Mono(公司版本)\Mono 的 background.js 整个文件只有一行，是混淆构建产物，不能在此改动，要找生成它的源码
- 允许例外：如果用户已经明确指出具体路径且愿意为此负责确认，可以直接采信用户指定的路径
- 置信度：中
- 用户动作/判断来源：2026-07-22 Mono 项目迁移：最初假设 D:\谷歌插件版本 下两份目录之一是要改的源码，实际上一份是 mock 参考副本（IMAGE2_PROVIDER="mock"），另一份是 build.js 混淆后的构建产物，真正在用的源码在 D:\workspace\claude\code\Mono 的 git company 分支
- 标签：源码定位, 多副本项目, git分支, 验证优先
- 待确认或待验证：无
- 最后更新日期：2026-07-22

### 规则名称：迁移资源前要确认所有实际访问方（尤其第三方云服务）的可达性

- 规则编号：待生成或待填写
- 状态：待验证
- 适用场景：把某个资源（存储、接口、下载地址等）从公网可达的位置迁移/替换为内网/私有网络地址时，尤其当这个地址不是只给自己的客户端用，还会被第三方服务（云函数、外部 API、SaaS 回调等）读取或访问
- 正向要求：动手迁移前，先列出这个地址实际会被谁访问——不只是浏览器/客户端/自己的服务器，还包括任何第三方云服务、外部 API 会不会主动发起请求去读这个地址；确认全部访问方都能到达新地址后再迁移，否则先保留旧的公网可达地址
- 应避免：不要只验证'我自己的客户端/浏览器能不能访问新地址'就判断迁移可行；接口层面（比如生成凭证、签名的 API 调用）测试通过，不代表下游真正使用这个地址的第三方服务也能访问到
- 判断依据：内网地址天然只对内网内的机器可达；任何运行在公有云/第三方机房的服务都无法访问私有网段地址，这是网络拓扑决定的硬限制，不是配置或代码可以绕过的问题（除非额外做公网穿透）
- 示例：
- Mono 视频上传预签名迁移：MinIO 在内网可以正常被浏览器 PUT 上传，但火山云的 video/analyze 云函数需要自己去下载视频，访问不到 192.168.1.198，导致视频分析失败
- 允许例外：如果确认所有访问方本来就都在同一个内网/私有网络里（没有第三方云服务参与），或者已经额外做好了公网穿透/反向代理让外部访问方也能到达，则不受此限制
- 置信度：中
- 用户动作/判断来源：2026-07-22 Mono 视频上传预签名迁移服务机：把视频存储从火山云 TOS 改到内网 MinIO 后，插件上传本身没问题，但火山云的 video/analyze 云函数要读视频时访问不到内网地址，导致整体功能失败
- 标签：基础设施迁移, 网络可达性, 第三方依赖, 架构判断
- 待确认或待验证：无
- 最后更新日期：2026-07-22

### 规则名称：内网服务要对外开放时优先做最小暴露面代理

- 规则编号：待生成或待填写
- 状态：待验证
- 适用场景：确实需要把一个原本只在内网使用的服务（对象存储、数据库、内部 API 等）开放给公网或第三方访问时
- 正向要求：优先在服务前面包一层窄接口/代理（比如只接受签名令牌、只做单一动作如下载指定对象），把这层代理暴露到公网；底层存储/数据库本身继续只在内网，不直接对外开放原生管理/数据 API
- 应避免：不要图省事直接把底层服务（MinIO 的 S3 API、数据库端口等）的端口整个映射到公网——即使当前用的凭据是最小权限、即使 URL 有签名保护，暴露面本身仍然更大（扫描器/爬虫可探测到该端口的存在，攻击面比一个定制的窄代理大）
- 判断依据：代理层可以精确限定'外部只能做什么'（比如只能下载已知 key 的对象），而直接暴露底层 API 意味着任何该 API 支持的操作理论上都在攻击面内，即使当前的凭据/策略把它们都拒绝了，仍然多一层不必要的风险和被扫描发现的机会
- 示例：
- Mono 视频下载：给 mono-server 新增 /download?key=&exp=&sig= 路由，服务器内部再去读 MinIO，MinIO 的 9000 端口本身不需要对公网开放
- 允许例外：如果暴露的资源本身设计上就是给公网直接访问的（比如公开的静态资源桶），或者时间/资源紧迫且已知只是临时方案，可以先用直接映射，但应明确标注为临时且尽快替换
- 置信度：中
- 用户动作/判断来源：2026-07-22 Mono 视频预签名迁移：需要让火山云能读到服务机上的视频时，在'直接把 MinIO 9000 端口映射到公网'和'mono-server 新增一个签名代理路由，只暴露下载这一个动作'之间，选择了后者
- 标签：安全, 对外暴露, 最小权限, 架构判断
- 待确认或待验证：无
- 最后更新日期：2026-07-22
