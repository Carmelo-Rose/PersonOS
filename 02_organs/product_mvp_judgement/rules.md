### 规则名称：前端底座阶段先补原生结构，不主动扩展产品层

- 规则编号：待生成或待填写
- 状态：待验证
- 适用场景：用户要求搭建前端底座、移植开源 UI 基础结构、复刻某个原生示例布局，且尚未要求完整产品信息架构或业务功能面板时。
- 正向要求：优先复用目标前端的原生结构和最小组合，补齐用户点名缺失的核心布局件，让底座先稳定可运行。
- 应避免：不要主动加入过多业务入口、状态标签、工具卡、工作台面板或品牌化信息架构，导致用户认为偏离底座任务。
- 判断依据：本次用户接受的是接近上一版的轻量 Base 体验，明确问题只在缺少 assistant-ui 原生侧边栏；过度扩展工具区反而被纠正。
- 示例：
- 正例：用 assistant-ui 的 ThreadListSidebar + Thread 搭出原生侧边栏和主聊天区，并保留 mock runtime 方便验证。
- 反例：在底座阶段加入图片反推、视频反推、Image2、素材下载的工具卡和自定义工作台面板。
- 边界例：用户明确进入产品化设计阶段后，才开始添加 Mono 自有工具入口和 workflow 面板。
- 允许例外：如果用户明确要求完整产品工作台、信息架构方案、或需要把现有业务 workflow 一起迁入，可以在底座之外增加产品层，但应说明这是下一层而非基础壳。
- 置信度：中
- 用户动作/判断来源：用户在 Mono assistant-ui 前端底座会话中纠正：不要加太多元素，先让前端底座搭建好，上一版挺好只是缺了原生侧边栏。, 用户此前指出完整组合不能只是中间聊天主模块，但随后明确边界是补原生侧边栏，而不是扩展工具工作台。
- 标签：frontend, MVP, scope-control, assistant-ui, Mono
- 待确认或待验证：该规则来自一次明确纠偏，后续同类前端底座任务需要继续验证。

### 规则名称：外部网页状态提示必须基于真实阻断而非静态文案

- 规则编号：待生成或待填写
- 状态：已验证
- 适用场景：产品依赖第三方网页、网页登录态、授权弹窗或外部播放器状态，并需要向用户解释卡顿、不可播放、不可控等问题。
- 正向要求：只有在用户目标被真实阻断时才提示外部网页原因；先观察播放进度、操作意图和错误状态，再结合登录/授权/弹窗证据分类提示。
- 应避免：不要仅因为页面存在登录按钮、试看文案、营销入口或静态 DOM，就提前显示“需要登录”之类的强提示。
- 判断依据：提示必须同时满足用户当前期望继续播放、播放进度连续不增长或媒体状态异常、且外部网页证据能解释该阻断；否则应保持安静或仅显示低干扰状态。
- 示例：
- 反例：B 站页面显示登录按钮和试看文案，但视频仍正常播放，此时显示“需要登录”属于误判。
- 正例：用户点击播放后，currentTime 连续一段时间不增长，同时原网页出现登录或验证弹窗，可以提示“播放停住，原网页可能需要登录或处理弹窗”。
- 边界例：currentTime 不增长但 readyState/networkState 显示缓冲，应提示缓冲而不是登录。
- 允许例外：安全、支付、权限等高风险阻断可以更早提示，但仍应说明依据；纯 UI 文案不足以作为阻断证据。
- 置信度：中
- 用户动作/判断来源：用户在 Web Audio Shelf 中指出：只要检测到 B 站登录文案就提示，会在播放正常时误判，要求撤回。, 用户要求遇到不确定产品问题先讨论，不要埋头开发；随后确认应以进度条/播放时间不动为前提，再检查原网页是否需要登录。
- 标签：产品判断, 外部依赖, 提示设计, 误判控制
- 待确认或待验证：具体停滞阈值需要根据真实用户测试继续调整。
- 最后更新日期：2026-07-09
