### 规则名称：无基础前端动效：分步构建优先

- 规则编号：待生成或待填写
- 状态：已验证
- 适用场景：自身缺乏前端基础，想用 AI 辅助生成复杂的前端动效（粒子系统、物理模拟、Canvas 动画等）
- 正向要求：必须先获取或制定分步路线图，将复杂效果拆为可独立验证的阶段，逐阶段构建并确认效果后再叠加下一层
- 应避免：避免用一段自然语言提示词描述完整的复杂动效并期望 AI 一次性生成可用的结果
- 判断依据：自然语言难以精确描述多层叠加的视觉效果；一次性生成时无法定位问题出在哪一层；分步构建每步都有明确的视觉验证标准
- 示例：
- 正面：按 8 步路线图逐阶段构建粒子特效，每步验证后叠加 → 成功
- 负面：找官网让 AI 反推提示词后直接丢给 Agent 生成完整效果 → 反复翻车
- 允许例外：对于简单效果（单个 CSS 动画、基础 DOM 交互）可直接生成；已有该领域丰富经验时可跳过分步
- 置信度：高
- 用户动作/判断来源：用户在粒子特效项目中亲身经历：一次性丢提示词给 Agent 生成复杂动效反复失败，改用分步构建后成功
- 标签：前端动效, AI辅助开发, 分步构建, 学习策略
- 待确认或待验证：无
- 最后更新日期：2026-06-21

### 规则名称：遇到硬性安全拦截时重新设计流程，不寻找替代手段绕过

- 规则编号：待生成或待填写
- 状态：已验证
- 适用场景：执行环境的安全分类器拦截某个具体动作（尤其明确标注硬规则、用户指令也无法解除时）
- 正向要求：退一步重新设计整个工作流，让被拦截的具体动作根本不需要发生，同时仍达成用户最终目的（如：密钥不做跨机传输，改为目标机本地填入；登录态不搬运，改为目标机重新登录；git push 改为用户本人执行）
- 应避免：换一个工具、参数、协议，或拆成多次小动作去达成和被拦截动作实质等价的效果；判断是否实质等价看最终效果（比如密钥内容是否离开了原机器），不看表面上换了什么手段
- 判断依据：拦截提示本身通常已经说明不要尝试绕过意图；本次迁移中 5 次拦截（git push master、git bundle --all 打包仓库、scp 传输 .env、scp 传输 browser_profile 登录态、读取远程 .env 内容调试）里，凡是重新设计流程的都顺利达成目标，说明换个方式绕过从来不是必要路径
- 示例：
- 正面：.env 改传非密钥模板+TODO占位，密钥由用户在目标机本地填入 -> 迁移成功，密钥全程未跨机器传输
- 正面：browser_profile 登录态不搬运，改为目标机重新登录 -> 更干净，避免设备指纹失效风险
- 负面（应避免的模式，本次未采用）：把 .env 内容读出来打印在别处、再手动誊写过去；把仓库拆成很多次小文件传输凑成整体迁移效果
- 允许例外：无——这是硬拦截场景，没有例外这个概念；真需要绕过时应该停下来向用户说明，让用户自己决定，而不是找技术手段绕
- 置信度：高
- 用户动作/判断来源：用户把罗盘项目从本机迁移到公司内网服务机的过程中，git push、git bundle --all、scp 传输 .env、scp 传输登录态、远程读取 .env 内容调试，先后 5 次被执行环境的硬性安全规则拦截, 每一次都没有尝试换工具/换协议/拆分小块绕过同一个动作，而是重新设计了达成目标的路径，最终都顺利完成迁移
- 标签：安全边界, 执行判断, 跨机器迁移, Agent协作
- 待确认或待验证：无
- 最后更新日期：2026-07-08

### 规则名称：学习项目第 0 阶段避免过度工程化

- 规则编号：待生成或待填写
- 状态：已验证
- 适用场景：用户以学习为主要目标启动一个技术项目，尤其是 Agent、AI 工具、前后端工作台等需要逐步理解核心机制的项目。
- 正向要求：第 0 阶段应优先建立最小可运行闭环：清晰目录、一个可验证接口、必要配置、基础测试、简短说明和少量学习注释；后续能力按学习路线逐步加入。
- 应避免：避免一开始把生产级模板当作必需品，包括过早加入 Docker、CI、数据库、复杂状态机、权限系统、MCP、向量库、完整前端设计体系或大量抽象基类。
- 判断依据：如果额外工程要求不能直接帮助用户理解当前阶段的核心概念，且会显著增加配置、依赖和排错成本，就应后置；如果它能形成可运行、可验证、可扩展的最小闭环，则可以保留。
- 示例：
- 正例：Agent Workbench Stage 0 保留 FastAPI、/api/health、配置集中管理、路由分层、Schema、pytest、React/Vite 状态页和 README。
- 反例：在尚未实现模型调用和 Agent Loop 前，就要求 Docker Compose、GitHub Actions、LangGraph、数据库、MCP、向量数据库和完整生产工程化。
- 边界例：前端不是必需项，但当用户希望看到后端状态时，可以手写一个轻量 Workbench 首页，并明确标记后续模块 Not implemented。
- 允许例外：如果项目目标本身是训练工程化、团队协作、部署上线或模板复用，可以在第 0 阶段提高工程完整度。
- 置信度：中
- 用户动作/判断来源：用户明确询问原开发提示词是否过于详细、限制是否过大。, 用户接受范围收敛建议，并要求按该判断执行 Agent Workbench 项目空壳开发。
- 标签：coding_judgement, scope-control, learning-project, agent, stage-0
- 待确认或待验证：无
- 最后更新日期：2026-07-13
