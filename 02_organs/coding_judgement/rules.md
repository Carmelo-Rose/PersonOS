### 规则名称：无基础前端动效：分步构建优先

- 规则编号：待生成或待填写
- 状态：已验证
- 适用场景：自身缺乏前端基础，想用 AI 辅助生成复杂的前端动效（粒子系统、物理模拟、Canvas 动画等）
- 正向要求：必须先获取或制定分步路线图，将复杂效果拆为可独立验证的阶段，逐阶段构建并确认效果后再叠加下一层
- 应避免：避免用一段自然语言提示词描述完整的复杂动效并期望 AI 一次性生成可用的结果
- 判断依据：自然语言难以精确描述多层叠加的视觉效果；一次性生成时无法定位问题出在哪一层；分步构建每步都有明确的视觉验证标准
- 示例：
- 正面：按 8 步路线图逐阶段构建粒子特效，每步验证后叠加 → 成功
- 负面：找官网让 AI 反推提示词后直接丢给 Agent 生成完整效果 → 反复翻车
- 允许例外：对于简单效果（单个 CSS 动画、基础 DOM 交互）可直接生成；已有该领域丰富经验时可跳过分步
- 置信度：高
- 用户动作/判断来源：用户在粒子特效项目中亲身经历：一次性丢提示词给 Agent 生成复杂动效反复失败，改用分步构建后成功；证据案例：success_20260621_001、success_20260624_001
- 标签：前端动效, AI辅助开发, 分步构建, 学习策略

### 规则名称：遇到硬性安全拦截时重新设计流程，不寻找替代手段绕过

- 规则编号：待生成或待填写
- 状态：已验证
- 适用场景：执行环境的安全分类器拦截某个具体动作（尤其明确标注硬规则、用户指令也无法解除时）
- 正向要求：退一步重新设计整个工作流，让被拦截的具体动作根本不需要发生，同时仍达成用户最终目的（如：密钥不做跨机传输，改为目标机本地填入；登录态不搬运，改为目标机重新登录；git push 改为用户本人执行）
- 应避免：换一个工具、参数、协议，或拆成多次小动作去达成和被拦截动作实质等价的效果；判断是否实质等价看最终效果（比如密钥内容是否离开了原机器），不看表面上换了什么手段
- 判断依据：拦截提示本身通常已经说明不要尝试绕过意图；本次迁移中 5 次拦截（git push master、git bundle --all 打包仓库、scp 传输 .env、scp 传输 browser_profile 登录态、读取远程 .env 内容调试）里，凡是重新设计流程的都顺利达成目标，说明换个方式绕过从来不是必要路径
- 示例：
- 正面：.env 改传非密钥模板+TODO占位，密钥由用户在目标机本地填入 -> 迁移成功，密钥全程未跨机器传输
- 正面：browser_profile 登录态不搬运，改为目标机重新登录 -> 更干净，避免设备指纹失效风险
- 负面（应避免的模式，本次未采用）：把 .env 内容读出来打印在别处、再手动誊写过去；把仓库拆成很多次小文件传输凑成整体迁移效果
- 允许例外：无——这是硬拦截场景，没有例外这个概念；真需要绕过时应该停下来向用户说明，让用户自己决定，而不是找技术手段绕
- 置信度：高
- 用户动作/判断来源：用户把罗盘项目从本机迁移到公司内网服务机的过程中，git push、git bundle --all、scp 传输 .env、scp 传输登录态、远程读取 .env 内容调试，先后 5 次被执行环境的硬性安全规则拦截, 每一次都没有尝试换工具/换协议/拆分小块绕过同一个动作，而是重新设计了达成目标的路径，最终都顺利完成迁移；证据案例：decision_20260708_001、success_20260716_001
- 标签：安全边界, 执行判断, 跨机器迁移, Agent协作
- 待确认或待验证：无

### 规则名称：可拖拽浮层的拖拽终止要依赖 pointer 生命周期

- 规则编号：待生成或待填写
- 状态：已验证
- 适用场景：实现浏览器内可拖拽浮层、折叠面板、侧边栏或其他需要稳定开始/结束状态的交互。
- 正向要求：拖拽状态必须由 pointer 生命周期管理，开始时锁定 pointerId 并启用 capture，结束时统一在 pointerup、pointercancel、lostpointercapture、window blur 等兜底路径清理状态。
- 应避免：不要仅依赖 mousemove、buttons、单点 mouseup 或鼠标当前位置来判断拖拽是否仍在继续；不要在拖拽过程中忽略失焦和捕获丢失。
- 判断依据：用户反馈的两类故障都说明按键状态本身是不可靠的终止信号；pointer capture 可以把拖拽行为绑定到同一 pointer 的完整生命周期，降低松手漂移和中途断连的概率。
- 示例：
- 正例：折叠助手面板拖拽时，pointerup 后立即停止移动，松手后鼠标再移动也不会带着面板走
- 正例：拖拽中切出窗口或发生捕获丢失时，lostpointercapture / blur 能保证状态及时清理
- 反例：只监听 mousemove 并读取 buttons，鼠标松开或系统切焦时容易进入错乱状态
- 允许例外：如果宿主环境没有 Pointer Events，或交互本身不是跨事件序列的拖拽，则可按简单方案处理，但要接受更弱的稳定性。
- 置信度：中
- 用户动作/判断来源：本次招聘助手折叠面板拖拽修复，用户指出松手后仍继续跟随鼠标、按住拖拽又偶发自动松掉, userscript/boss-helper.user.js 中将拖拽收尾改为 pointerup / pointercancel / lostpointercapture / blur
- 标签：????, drag, pointer-events, ui-reliability
- 待确认或待验证：无
- 最后更新日期：2026-07-17
