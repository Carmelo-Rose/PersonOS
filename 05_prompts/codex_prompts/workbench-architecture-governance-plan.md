# Workbench 架构审计与分阶段重构规划提示

## 元数据

- 状态：实验中
- 分类：codex_prompts
- 工具或场景：Codex/Claude Code 对 AI Workbench 做架构审计、回归定位和分阶段重构规划
- 版本：v1
- 最后更新：2026-07-24
- 标签：架构审计, 分阶段重构, 回归测试, Codex, Claude Code

## 用户动作/判断来源

- 用户明确要求审查并治理 Workbench 分层架构
- 本次确认的架构治理决策

## 输入要求

- 仓库路径和运行方式
- 入口清单及复现步骤
- 环境变量名与 provider 配置（脱敏）
- 现有测试和日志

## 提示词正文

```text
你是负责架构治理的工程师。先只读审计项目：绘制聊天、工具、MCP/扩展入口到能力模块、素材存储、provider 和异步任务的依赖图，找出重复业务逻辑、跨层调用、隐式 fileId/Base64 协议和 Web 进程承担长任务的位置。先补 characterization、入口契约、asset/provider 请求和 Worker 恢复测试，记录当前行为与可回滚点。然后给出分阶段迁移计划：工具入口直接执行类型化业务命令，聊天入口解析为同一命令；统一逻辑 assetId；provider 只通过 ports/adapters；Web 只持久化 job，独立 Worker 执行。保留现有 API/MCP/扩展兼容适配器。每阶段列出改动边界、验收测试、风险、回滚方式和不应同时做的事项。不要继续在旧路由上堆补丁，不要一次性引入新的分布式基础设施。
```

## 输出格式

- 依赖图和分层违规清单
- 按阶段的迁移表
- 契约/回归测试矩阵
- 风险与回滚清单
- 待用户确认的唯一决策

## 成功标准

- 聊天和工具入口最终共享同一业务命令
- 素材使用统一逻辑 assetId
- provider 与业务模块隔离
- 长任务不阻塞 Web 进程
- 旧兼容入口可验证且每阶段可回滚
- 没有新增跨层特判

## 验证结果

尚未在本 Workbench 仓库执行；本次已由用户确认作为后续审计与重构的工作提示。

## 适用边界

- 适用于已有多入口、多模型和异步任务的 AI Workbench
- 不替代具体 provider 的官方请求体验证
- 如代码库很小，应保留原则但按最小模块实现

## 变更原因

把一次架构治理讨论固化为可复用的审计和规划提示，减少后续需求继续形成屎山式增量耦合。
