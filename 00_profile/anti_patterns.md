## 反模式记录模板

### 反模式名称：把模糊指令丢给弱模型 / 子代理

- 适用场景：用 AI 子代理或弱模型做设计 / 开发
- 典型表现：只给风格名或模糊目标 → 产出换皮模板、被评“都是垃圾”；弱模型直接开发结果不可控（“一坨屎”）
- 为什么有害：返工成本高，浪费 token 与时间；且执行体容易交付前不自评
- 早期信号：指令里只有抽象风格词、没有具体参考 / 约束；交付前未做质量自评
- 替代做法：先给具体参考（URL / 截图）+ 从身份 / 概念出发 + 写明约束；强模型定架构、先写 SPEC 再落地
- 允许例外：纯探底 / 试错（已明确预期效果不佳，仅用于摸边界）
- 来源案例：failure_20260619_001、decision_20260620_001、project_20260618_001、project_20260620_001
- 最后复核日期：2026-06-20

### 反模式名称：凭感觉判断根因，不先最小实测

- 适用场景：排查性能、卡死、数据异常、环境或时区问题的根因时
- 典型表现：见长时间无响应就断定“卡死”，凭经验直接下网络/配置结论
- 为什么有害：误判方向、反复返工；把本可量化的问题当成玄学
- 早期信号：还没跑 curl 压测 / dry-run / 日志 UTC 锚点，就已经有了根因结论
- 替代做法：先用最小实测或量化手段拿到确定数据（真实 tok/s、真实阻断、时区差值），再下结论
- 允许例外：已有充分同类经验、验证成本极低的显而易见问题
- 来源案例：failure_20260703_001、decision_20260704_001、decision_20260708_001
- 最后复核日期：2026-07-16
