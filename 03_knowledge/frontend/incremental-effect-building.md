# 复杂前端动效的渐进式构建法

## 元数据

- 类型：方法

## 用户动作/判断来源

- 用户在粒子特效项目中对比两种方式：一次性让 AI 生成完整效果反复失败，改用分步构建后成功。
- 用户在 Premium Cursor Mask / MiMo Ink Reveal session 中明确提出，不想每做一个页面就另起一个项目，因为麻烦且容易忘记。
- 本次在原 particle-logo-demo 内新增 MiMoInkReveal，并把 App 改为 hash route + lazy component 的动效实验台，保留原粒子页和其他 reveal demo。

## 适用条件

- 需要实现复杂 Canvas/WebGL/CSS 前端动效。
- 自身缺乏前端基础或对该类动效不熟悉。
- 使用 AI 辅助生成代码时。
- 同一主题下会持续试验多个视觉动效、路线或参考站复刻。

## 核心内容

- 复杂前端动效应拆为独立可验证的阶段逐层叠加，而非一次性让 AI 生成完整效果。每个阶段应有明确的视觉验证标准，确认当前阶段效果满意后再叠加下一阶段。
- 当多个动效属于同一探索主题时，应优先沉淀在同一个实验台项目中，通过 route、hash、tab 或 story 管理，而不是每个页面另起一个项目。这样可以复用依赖、样式、验证命令和本地运行方式，降低启动摩擦和遗忘成本。
- 对 Vite/React 这类轻量 demo 项目，可以用 hash route 管理效果：#/ 保留主效果，#/ink、#/mask、#/morph 等挂载独立组件；重型 three/WebGL demo 使用 lazy import，避免所有效果进入首包。
- 一次性生成方式的问题在于：自然语言难以精确描述复杂视觉效果，调试时无法定位是哪一层的参数出了问题；拆分 route/component 后，每个效果可以单独验证和对比。

## 边界与例外

- 对于完全不同产品、不同设计系统或不同运行环境的页面，另起项目仍可能更清晰。
- 实验台项目需要保持入口清晰，避免把导航、依赖和全局样式弄成新的复杂度来源。
- 简单 CSS 动画或一次性静态页面无需过度路由化。

## 失效风险

低。渐进式构建和同项目实验台是通用工程组织原则，不依赖单一前端框架；具体路由方式会随项目技术栈变化。
