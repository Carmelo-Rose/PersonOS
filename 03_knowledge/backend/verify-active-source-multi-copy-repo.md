## 适用条件

- 同一个项目在文件系统里有多个同名/相似目录（不同工具的工作副本、历史备份、构建产物、不同分支导出）
- 需要确认哪一份是当前实际部署/在用的版本再动手修改

## 核心内容

- 先看是不是 git 仓库：git branch -a 看有哪些分支，git status/git log 确认当前签出的分支和是否干净
- 看内容特征而不是文件名/目录名：关键配置项是不是明显的占位/mock 值（如 IMAGE2_PROVIDER="mock"），还是真实的第三方 provider 值
- 看文件形态：源码应该是可读的多行代码；如果整个文件被压缩成一行且变量名是乱码（_0x 开头等），大概率是混淆后的构建产物，不是源码，不能在这上面改
- 交叉验证：用真实数据反向核对，比如从线上导出的真实数据结构（字段丰富程度）和某份代码实现的字段是否对得上，对不上说明这份代码不是当前线上版本

## 边界与例外

- 用户已经明确指定路径并确认负责时，可以直接采信，不必每次都重新排查
- 此方法假设本地文件系统里能同时看到多个候选目录；如果只有唯一一份代码，不需要这套流程

## 失效风险

如果项目改用单一代码库+CI/CD 部署（不再有多目录并存），这套排查方法的必要性会降低；具体特征信号（mock 标记、混淆检测）也要随项目实际情况调整
