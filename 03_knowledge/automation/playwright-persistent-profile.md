# Playwright persistent profile 互斥锁与登录态诊断方法

## 元数据

- 状态：已验证
- 类型：方法
- 领域：automation
- 最后更新：2026-07-04
- 标签：Playwright, 浏览器自动化, 资源互斥, 诊断方法, 罗盘项目

## 用户动作/判断来源

- 用户在罗盘(luopan)项目里想新增一个独立的商品属性抓取脚本，追问“能不能用独立 profile 规避跟主任务的冲突”
- 本次会话做了对比实测：分别用已登录的罗盘 profile 和一个全新未登录的临时 profile 打开同一个抖音商品详情 URL，对比结果差异来验证结论

## 适用条件

- 用 Playwright launch_persistent_context 复用登录态做采集/自动化时，评估要不要新增独立脚本、是否需要错峰运行
- 遇到“能不能用一个全新/独立环境规避跟现有任务的资源互斥”这类问题

## 核心内容

- launch_persistent_context(user_data_dir=...) 对同一个 profile 目录是独占锁，同一时刻只能被一个 Chrome 进程持有；两个脚本指向同一目录会冲突报错，指向不同目录则完全独立、可并发
- 冲突范围是“任务从触发到真正跑完的整个运行区间”，不是只避开触发那一瞬间
- 遇到“能否用独立环境规避互斥”的问题，优先做最小实测而不是凭经验判断：用一个全新、从未登录的临时 profile 打开同一目标页面，看是否被拦截，能快速验证该资源是否绑定登录态。本次实测中，抖音罗盘详情页对未登录 profile 返回“登录已过期”拦截页，说明该页面依赖登录态，无法靠独立无登录 profile 绕开

## 边界与例外

- 只有目标资源确实依赖登录态时，“复用同一 profile”才是唯一选择；若目标资源不需要登录态，独立/临时 profile 完全可以用来规避互斥
- 结论基于当前罗盘详情页的登录校验行为，站点改版或策略变化需要重新验证

## 失效风险

中，依赖目标站点当前的登录校验策略
