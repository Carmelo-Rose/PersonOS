## 用户动作/判断来源

- 用户在 auto_pdd 项目里确认该应用的目标是从飞书待发布记录一键生成拼多多草稿，并追问最小使用流程与技术栈归类。
- 当前会话中结合 lark_bridge.py、publish_worker.py、ai_generate.py、pdd_fill.py、console_server.py 的代码结构，对链路做了静态核对与归类。

## 适用条件

- 需要把表格型运营数据作为任务源，在本机浏览器登录态下驱动电商后台保存草稿。
- 需要在不直接开放平台 API 的前提下，用本地自动化把飞书多维表格、AI 识别和后台表单填写串起来。

## 核心内容

- 这类链路可以拆成六层：Python 编排层、飞书 Base/lark-cli 数据桥接层、DashScope 图片识别层、Chrome CDP 表单自动化层、本地 Web 控制台层、文件队列与日志层。
- 其中飞书负责候选任务与状态回写，AI 负责把图片识别成拼多多可用的类目/标题初稿，真正依赖人工登录态和页面行为的是 Chrome CDP 草稿保存。
- 如果目标是低风险上线，自动化终点应停在保存草稿，人工再审核提交；如果要规模化和高稳定，最终应考虑平台官方 API 替代页面自动化。

## 边界与例外

- 不适用于完全无人工登录态、无浏览器环境的纯云端任务。
- 不应把这条知识误解为“可以安全自动发布”，因为当前架构更适合停在草稿。
- 当平台已有稳定开放接口时，这种页面自动化架构只应作为过渡方案。

## 失效风险

中。飞书字段结构、DashScope 模型接口、拼多多商家后台 DOM 和 Chrome 调试方式都可能变化。
