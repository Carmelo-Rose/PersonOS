# 主动沉淀提醒应用 Hook，写入继续交给 Skill

## 元数据

- 状态：已验证
- 类型：架构判断
- 领域：ai_agent
- 最后更新：2026-06-17
- 标签：PersonOS, Skill架构, Hook, 主动提醒, 会话采集, AI Agent

## 用户动作/判断来源

- 用户追问当前 skill 是否能在对话过程中主动提醒适合沉淀的内容。
- 结合 Claude 官方关于 skills 自动加载与 hooks 生命周期触发的说明，用户确认需要区分偶发提醒与确定性提醒。
- 用户据此把 PersonOS 方向收敛为：提醒层与写入层分离，hook 或外层事件机制负责提醒，session-capture 与专项 capture skill 负责整理、路由和写入。

## 适用条件

- 希望 Agent 在对话过程中及时提示哪些内容适合沉淀到 PersonOS。
- 需要把提醒能力与写入能力拆开，避免把所有职责都压进一个主 skill。
- 需要比模型临场判断更稳定的提醒触发机制。

## 核心内容

- Skill 可以因为描述匹配而被模型自动加载，也可能在对话中顺势提醒“这段适合记录”，但这种提醒本质上依赖模型判断，不是确定性监听。
- 如果目标是稳定的主动提醒，应把检测放在 hook 或外层事件机制上，例如在每轮用户输入或每轮停止时检查最近上下文，再决定是否提示记录。
- 写入流程仍应保持在 session-capture 与各专项 capture skill 中：主 skill 负责整理上下文、选择目标组、生成 handoff，专项 skill 负责新建/更新判断、草稿、确认和落库。
- 最小闭环是：hook 或事件层检测到可沉淀信号 -> 给出提醒 -> 用户确认 -> session-capture 组织当前会话 -> 对应专项 skill 写入。

## 边界与例外

- 不要让 hook 直接越过确认门槛写入 PersonOS；提醒层和写入层应继续分离。
- 没有 hooks 或事件机制的环境里，只能退回到 skill 的弱提醒，不能承诺每次都及时提示。
- 要限制提醒频率，避免每轮对话都触发提醒造成噪音。

## 失效风险

依赖目标 Agent 是否提供 hooks、事件回调或类似生命周期能力；如果只有普通聊天窗口而无事件机制，这条方法只能部分成立。

## 待确认或待验证

- 不同 Agent 对 hooks 或外层事件机制的可用性与稳定性仍需分别验证。
- 提醒阈值、频率控制与去重策略还没有收敛成固定方案。
