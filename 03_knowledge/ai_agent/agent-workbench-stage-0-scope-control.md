# 学习型 Agent 项目的 Stage 0 应收敛为最小可运行骨架

## 用户动作/判断来源

- 用户以边学边做 Agent 为目标，质疑原第 0 阶段提示词包含 Docker、CI、前端、后端、文档、Lint 等完整工程化要求，认为限制条款过多。
- 用户接受将第 0 阶段收敛为最小可运行骨架，并实际完成 FastAPI 后端、React/Vite 前端、健康检查、模块占位和项目记录。

## 适用条件

- 开发目标是学习 Agent 核心能力，而不是一开始交付生产级平台。
- 项目处于第 0 阶段或原型阶段，需要先建立能跑、能测、能扩展的基本结构。
- 用户需要通过实际开发理解模型调用、Tool Calling、Agent Loop 等后续能力。

## 核心内容

- 第 0 阶段的关键不是一次性搭完整工程模板，而是形成最小闭环：后端能启动、前端能访问、健康检查可验证、目录结构支持后续扩展。
- 对学习型 Agent 项目，应把 LangGraph、MCP、数据库、Memory、RAG、Docker、CI、权限系统等后置，避免学习重心从 Agent 核心机制转移到工程杂项。
- 保留少量工程习惯是有价值的：配置集中管理、路由分层、Schema、测试、简短中文注释、README。它们能帮助后续扩展，但不会显著增加认知负担。
- 前端可以在需要可视化状态时加入，但应保持为真实可用的工作台首页，而不是营销页或假装完整平台。

## 边界与例外

- 如果目标是团队协作或上线交付，Docker、CI、Lint、权限、部署等工程化要求应提前规划。
- 如果用户明确要练习工程化能力，第 0 阶段可以提高脚手架完整度。
- 最小骨架不是把所有文件堆在一个文件里；仍需要保持基本分层和可测试性。

## 失效风险

低。作为学习项目范围控制原则较稳定；但当项目进入生产工程化阶段时，需要重新引入 Docker、CI、部署、权限、评测等能力。
