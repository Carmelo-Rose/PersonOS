# 04_cases

存放跨项目的结构化案例，包括项目、决策、失败和成功案例。案例必须区分事实、判断与复盘，不确定内容应明确标记。

## 核心准则

**PersonOS 案例必须以用户本人为主体；Agent、工具与技术过程仅作为背景、证据或结果。**

自检：删掉所有 Agent 名称后，仍应清楚看出用户的行动、判断与成长。

维护时优先补充真实结果与可复用经验。后续可按类型、标签、背景和结果检索，供 Agent 类比推理、RAG 和评测使用。


## 决策案例的两条正交状态轴

`decision_cases.jsonl` 每条决策有两个互不相关的状态字段，检索/RAG/评测/微调时应分别对待：

- **可信度轴 `状态`**：这条记录本身可不可信。取值：`示例/占位`、`真实/待验证`、`真实/已验证`。
- **生命周期轴 `决策状态`**：这条决策现在还算不算数。取值：`现行`、`已取代`、`已废弃`、`提议中`、`示例`。

配套两个取代指针（默认空字符串）：

- `取代`：本条取代了哪条决策的编号（写在**新**决策上）。
- `被取代`：本条被哪条决策取代的编号（写在**旧**决策上）。

**维护约定**：当某决策被新决策替换时，成对更新——新决策 `决策状态:现行` + `取代:<旧编号>`，同时把旧决策改为 `决策状态:已取代` + `被取代:<新编号>`。旧决策**不删除**，保留当时的背景与理由。

**给下游用**：RAG/Agent 检索决策时应优先只认 `决策状态=现行`；命中 `已取代` 时顺 `被取代` 指针跳到现行决策，避免复用已被否定的判断。微调取语料时同理，排除非现行决策以免信号自相矛盾。
